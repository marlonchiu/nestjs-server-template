## 1. 概述

### 1.1 背景

当前 NestJS 服务框架仅有 NestJS 内置的基础 Logger（`Logger` 类），存在以下问题：

1. **无结构化**：纯文本日志，难以解析和检索
2. **无持久化**：仅输出到 console，生产环境无法回溯
3. **无轮转机制**：日志文件无限增长，缺乏磁盘管理
4. **无分类能力**：系统日志和业务日志混杂，难以区分
5. **无上下文**：无法追踪请求链路（traceId）

### 1.2 目标

建立统一的日志服务，满足以下需求：
- 日志结构化输出（JSON 格式）
- 日志持久化到文件系统
- 日志按日期/大小自动轮转
- 支持日志分类管理
- 支持请求链路追踪
- 可扩展至日志查询 API

### 1.3 成功指标

- [ ] 应用启动时自动初始化日志服务
- [ ] 所有 API 请求自动记录访问日志
- [ ] 错误自动记录错误日志并包含堆栈信息
- [ ] 日志按日期自动分割存储
- [ ] 日志文件支持自动压缩和清理
- [ ] 支持通过配置控制日志级别

## 3. 功能需求

### FR-001：统一日志服务封装
- **描述**：封装统一的日志服务，提供 `LoggerService` 替代 NestJS 内置 Logger
- **优先级**：必须有
- **验收标准**：
  - [ ] 提供 `LogsModule` 模块
  - [ ] 提供 `LogsService` 服务
  - [ ] 支持注入到其他模块使用
  - [ ] 支持自定义日志上下文

### FR-002：结构化 JSON 日志
- **描述**：日志以 JSON 格式输出，包含标准化字段
- **优先级**：必须有
- **依赖**：FR-001
- **验收标准**：
  - [ ] 日志包含字段：`timestamp`, `level`, `message`, `context`, `traceId`（可选）
  - [ ] 错误日志包含 `stack` 字段
  - [ ] 可配置是否开启 JSON 格式

### FR-003：日志文件持久化
- **描述**：日志同时输出到文件和控制台
- **优先级**：必须有
- **依赖**：FR-001
- **验收标准**：
  - [ ] 配置日志输出目录（默认 `./logs`）
  - [ ] 分别输出到不同日志文件：
    - `app.log` - 应用日志
    - `error.log` - 错误日志（仅错误级别）
    - `access.log` - 访问日志
  - [ ] 控制台保留彩色输出（开发环境）

### FR-004：日志轮转机制
- **描述**：日志文件按日期或大小自动轮转
- **优先级**：应该有
- **依赖**：FR-003
- **验收标准**：
  - [ ] 按天分割日志文件（`app.2026-03-18.log`）
  - [ ] 支持配置最大文件大小
  - [ ] 支持自动压缩历史日志（`.gz`）
  - [ ] 支持配置日志保留天数

### FR-005：访问日志拦截器
- **描述**：自动记录所有 HTTP 请求访问日志
- **优先级**：应该有
- **依赖**：FR-001
- **验收标准**：
  - [ ] 自动拦截所有 API 请求
  - [ ] 记录内容：请求方法、URL、状态码、响应时间、客户端 IP、User-Agent
  - [ ] 支持配置排除路径（如健康检查）

### FR-006：错误日志增强
- **描述**：全局异常过滤器自动记录错误日志
- **优先级**：应该有
- **依赖**：FR-001
- **验收标准**：
  - [ ] 自动捕获所有未处理异常
  - [ ] 记录完整堆栈信息
  - [ ] 记录请求上下文（URL、方法、参数）

### FR-007：请求链路追踪
- **描述**：为每个请求生成唯一追踪 ID
- **优先级**：应该有
- **依赖**：FR-005
- **验收标准**：
  - [ ] 自动为每个请求生成 `traceId`
  - [ ] `traceId` 可传递到子日志
  - [ ] 可通过响应头返回 `traceId`

### FR-008：日志级别管理
- **描述**：支持运行时配置日志级别
- **优先级**：应该有
- **依赖**：FR-001
- **验收标准**：
  - [ ] 支持级别：debug, info, warn, error
  - [ ] 可通过 .env 配置全局级别
  - [ ] 可按模块/上下文配置级别

## 4. 非功能需求

### NFR-001：性能影响
- **类型**：性能
- **描述**：日志写入不应显著影响请求响应时间
- **指标**：异步写入，引入延迟 < 5ms

### NFR-002：日志可读性
- **类型**：可维护性
- **描述**：开发环境下日志应保持可读性
- **指标**：开发环境控制台彩色输出，生产环境 JSON 格式

### NFR-003：磁盘空间管理
- **类型**：运维
- **描述**：防止日志无限增长导致磁盘空间耗尽
- **指标**：默认保留 7 天日志，自动清理过期文件

### NFR-004：类型安全
- **类型**：代码质量
- **描述**：TypeScript 严格模式，无 any 类型泄漏
- **指标**：tsconfig 开启 strictNullChecks

## 5. 技术设计

### 5.1 技术选型

| 组件 | 选项 | 选择 | 原因 |
|------|------|------|------|
| 日志库 | Winston / Pino | **Pino** | 高性能、JSON 原生、NestJS 官方推荐 |
| 轮转库 | pino-rotate / winston-daily | **pino-rotate** | 配合 Pino 使用简单 |
| 文件写入 | fs / stream | **stream** | 内存效率更高 |

### 5.2 目录结构

```
src/
├── modules/
│   └── logs/                    # 新增日志模块
│       ├── logs.module.ts       # 日志模块
│       ├── logs.service.ts     # 日志服务
│       ├── logs.interceptor.ts # 访问日志拦截器
│       ├── logs.filter.ts      # 错误日志过滤器
│       ├── logs.middleware.ts  # 请求追踪中间件
│       └── entities/           # 日志实体（可选：持久化到数据库）
│           └── access-log.entity.ts
```

### 5.3 日志格式

**JSON 输出示例**：
```json
{
  "level": 30,
  "time": "2026-03-18T10:00:00.000Z",
  "pid": 12345,
  "hostname": "server-01",
  "traceId": "abc-123-def",
  "context": "AuthService",
  "msg": "User logged in",
  "userId": "user-uuid",
  "ip": "192.168.1.1"
}
```

**字段说明**：
| 字段 | 类型 | 说明 |
|------|------|------|
| level | number | 日志级别 (20=debug, 30=info, 40=warn, 50=error) |
| time | string | ISO 时间戳 |
| pid | number | 进程 ID |
| hostname | string | 主机名 |
| traceId | string | 请求追踪 ID |
| context | string | 日志上下文（模块/类名） |
| msg | string | 日志消息 |
| userId | string | 用户 ID（可选） |
| ip | string | 客户端 IP（可选） |
| stack | string | 错误堆栈（仅错误日志） |

### 5.4 环境配置

```bash
# 日志配置
LOG_LEVEL=info              # 日志级别: debug, info, warn, error
LOG_DIR=./logs             # 日志目录
LOG_MAX_SIZE=100m          # 单文件最大大小
LOG_MAX_FILES=7            # 保留文件数（天）
LOG_JSON=true              # JSON 格式输出（生产环境）
LOG_ACCESS=true            # 记录访问日志
```
